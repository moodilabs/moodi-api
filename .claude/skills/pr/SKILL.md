# PR 생성 스킬

## 설명

현재 브랜치의 변경사항을 분석하여 PR을 생성한다.

PR은 코드 변경 목록이 아니라 **기능과 구현 의도를 설명하는 문서**로 작성한다.

리뷰어가 git diff를 보지 않아도 변경 목적과 영향 범위를 이해할 수 있도록 작성한다.

## 참고 파일

- `CLAUDE.md` — 프로젝트 컨벤션 및 아키텍처
- `.github/pull_request_template.md` — PR 템플릿 원본

## 절차

1. `git status`로 작업 상태 확인
2. `git log main..HEAD --oneline`으로 커밋 확인
3. `git diff main...HEAD`로 변경사항 분석
4. 변경사항을 **기능 단위**로 재구성
5. PR 제목 작성
6. PR 본문 작성
7. 사용자에게 제목/본문 확인받기
8. 승인 후 `gh pr create`

## PR 제목 규칙

- 커밋 컨벤션을 따른다: `feat:`, `fix:`, `refactor:`, `docs:`, `chore:`, `test:`, `style:`
- 70자 이하
- 변경사항의 핵심 기능을 제목으로 작성
- 커밋 제목을 그대로 복사하지 않는다


## PR 작성 원칙

### 작업 내용

작업 내용은 단순히 변경사항을 나열하지 않는다. 다음 순서를 따른다.

1. **왜** 이 작업을 하게 되었는지
2. **어떤 방식**으로 해결했는지
3. 리뷰어가 알아야 하는 **구현상의 판단이나 영향**

2~4문단 정도의 자연스러운 설명으로 작성한다.

가능한 경우 아래 내용을 포함한다.

- 작업 배경
- 해결 방법
- 설계 선택 이유
- 운영 환경 영향
- 후속 작업(있다면)


### 변경사항

변경사항은 **파일 단위가 아니라 기능 단위**로 작성한다.


### API 스펙 (REST API 추가/변경 시)

REST API가 추가되거나 변경된 경우에만 작성한다. 내부 리팩토링, 설정 변경, 인프라 작업에서는 작성하지 않는다.

| 항목 | 값 |
|---|---|
| Method | `GET` / `POST` ... |
| Path | `/api/v1/...` |
| 파라미터 1 | 타입, 필수/선택, 제약 |
| ... | ... |

시나리오별 응답:

**정상** — `요청 예시`
```json
HTTP 200
{ ... }
```

**검증 실패** — `요청 예시`
```json
HTTP 400
{ "error": { "message": "..." } }
```

### 테스트 결과

실제로 수행한 검증만 작성한다. 

- [x] `./gradlew test` 통과 (N tests, 0 failed)
- [x] [수동 검증 항목]
- [ ] [리뷰어가 검증할 항목]

### 관련 이슈


## 작성 규칙

1. PR 제목은 커밋 컨벤션을 따른다 (feat:, fix:, chore: 등)
2. 변경사항을 커밋 단위가 아니라 **기능 단위**로 정리한다.
3. 작업 내용은 변경 이유와 구현 방식을 설명한다. 단순 요약만 작성하지 않는다.
4. 리뷰어가 확인해야 하는 내용이 있다면 작업 내용에 포함한다.
5. 파일 변경 목록을 그대로 나열하지 않는다. 반드시 기능 중심으로 작성한다.
6. REST API 변경이 없는 경우 API 스펙/응답 예시 섹션은 생략한다.
7. 본문은 **리뷰어를 위한 문서**라는 관점으로 작성한다. git diff를 읽기 전에 전체 변경사항을 이해할 수 있도록 작성한다.
8. base 브랜치는 `main`을 기본으로 한다.
